# LX0701: ImageButton lacks accessibility description

<!-- Upstream: RXT351. -->

**Applies to:** `MAUI`

## Cause

A MAUI `<ImageButton>` has no attribute that describes it for assistive
technologies, and is not explicitly marked as decorative.

## Rule description

`ImageButton` is a tappable control whose only visible content is an image.
Screen readers cannot tell the user what the button does — "unnamed image
button" is all they can announce — so the author must provide a text
alternative through MAUI's `AutomationProperties` attached properties, or
explicitly mark the control as outside the accessibility tree.

**The rule is off by default in `:recommended`** because most codebases
have not done a deliberate accessibility pass; enabling it there would
flood output. Teams who want to enforce a11y should extend `:strict`, or
explicitly set `LX0701: warning` in their config.

**Suppressors:**

- `AutomationProperties.Name` — concise name (typically a verb, e.g. "Close").
  Any value (literal or bound) suppresses.
- `AutomationProperties.HelpText` — longer description. Any value
  (literal or bound) suppresses.
- `SemanticProperties.Description` — MAUI-idiomatic short AT name. Any
  value (literal or bound) suppresses.
- `SemanticProperties.Hint` — MAUI-idiomatic longer AT hint. Any value
  (literal or bound) suppresses.
- `AutomationProperties.LabeledBy` — reference to another element that
  provides the label. Two statically-resolvable forms are recognised
  and scope-validated identically: `{x:Reference <name>}` (XAML 2009
  primitive) and `{Binding ElementName=<name>}` (classic WPF idiom —
  rare on MAUI but syntactically valid). The target must be declared
  in the same XAML name scope as the ImageButton; a dangling
  reference does not suppress. Other markup extensions
  (`{Binding Path=…}` without `ElementName`, `{StaticResource …}`,
  etc.) suppress permissively because static evaluation isn't
  possible. Non-extension literals suppress on any non-empty value.
- `AutomationProperties.IsInAccessibleTree="False"` — explicit decorative
  opt-out (rare for `ImageButton`, since a control without a label is
  usually a design bug). Only a literal `"False"` (case-insensitive) or a
  bound value suppresses; `"True"` reasserts default inclusion and
  therefore does not suppress.
- `AutomationId` — the canonical test-automation hook on UI Automation
  (Windows) and MAUI's `IElement.AutomationId`. Any value (literal or
  bound) suppresses.

```xaml
<!-- non-compliant: no a11y info -->
<ImageButton Source="icon.png" />
```

```xaml
<!-- compliant: verb-first name -->
<ImageButton Source="icon.png" AutomationProperties.Name="Close" />
```

## Known limitation

The rule does not verify the element referenced by
`AutomationProperties.LabeledBy="{x:Reference …}"` has user-visible text
(e.g. a `Label.Text` or adjacent caption). A reference to a layout
`<Border/>` passes the check even though it's nonsensical for an AT.
Tracked separately; the name-resolution infrastructure is in place but
type-aware validation is not.

## How to fix violations

Add `AutomationProperties.Name="…"` with a verb-first label that names the
button's action ("Close", "Add to cart", "Open menu"). If an adjacent Label
already names the button, use
`AutomationProperties.LabeledBy="{x:Reference LabelName}"` to reuse it.

Marking an `ImageButton` decorative with `IsInAccessibleTree="False"` is
rare — a button with no label is usually a design bug — so prefer the
name/help-text path.

Test the fix by running a screen reader (TalkBack on Android, VoiceOver
on iOS) and confirming the announcement makes sense.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX0701 -->
<ImageButton Source="icon.png" />
```

For a whole file or project:

```json
{ "rules": { "LX0701": "off" } }
```
