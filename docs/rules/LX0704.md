# LX0704: Icon button lacks accessibility description

**Applies to:** `Avalonia`  `MAUI`  `Uno`  `UWP`  `WinUI 3`  `WPF`

## Cause

A `<Button>` (or MAUI `<ImageButton>`) whose visible content is an icon,
glyph, or punctuation symbol — or which is empty — carries no attribute
that describes it for assistive technologies.

## Rule description

An icon-only button is opaque to a screen reader: there's no text the AT
can read out, so the user hears at best "button" with no indication of
what tapping it will do. The author must provide a text alternative
through the dialect's accessibility attached properties, reuse a nearby
label via `AutomationProperties.LabeledBy`, or — rarely — mark the
control as outside the accessibility tree.

The rule recognises an icon-only button in three shapes:

- **Symbolic `Content` attribute.** When `Content="…"` contains only
  non-letter, non-digit runes, the value is treated as chrome — UI
  punctuation like `"+"`, `"-"`, `"…"`, `"→"`, or a Unicode Private
  Use Area glyph from an icon font (Segoe MDL2 Assets, Segoe Fluent
  Icons, Material Icons, FontAwesome, …). PUA code points are not
  classified as letters or digits by Unicode category, which is what
  makes the heuristic work without an icon-font allowlist. The same
  helper backs LX0400's symbol-skip logic — the two rules read symbolic
  content identically.
- **Single icon-element child.** When the button has exactly one
  layout child and that child is an icon-class element. Property-element
  syntax (`<Button.Style>`, `<Button.Resources>`) is not counted as
  layout content. The recognised icon-class elements span dialects:

  | Element | Dialects |
  |---|---|
  | `Image` | All |
  | `FontIcon` | WinUI 3, UWP, Uno |
  | `SymbolIcon` | WinUI 3, UWP, Uno |
  | `PathIcon` | WinUI 3, UWP, Uno |
  | `BitmapIcon` | WinUI 3, UWP, Uno |
  | `ImageIcon` | WinUI 3, UWP, Uno (newer SDKs) |
  | `Path` | WPF, Avalonia (canonical icon idiom) |

- **Empty button.** A `<Button />` with no `Content` attribute and no
  layout children — a placeholder a screen reader cannot describe.

**The rule is off by default in `:recommended`** because most codebases
have not done a deliberate accessibility pass; enabling it there would
flood output. Teams who want to enforce a11y should extend `:strict`, or
explicitly set `LX0704: warning` in their config.

**Suppressors** (identical to LX0700–LX0703):

- `AutomationProperties.Name` — concise verb-first name. Any value
  (literal or bound) suppresses.
- `AutomationProperties.HelpText` — longer description. Any value
  (literal or bound) suppresses.
- `SemanticProperties.Description` — MAUI-idiomatic short AT name.
  Any value (literal or bound) suppresses.
- `SemanticProperties.Hint` — MAUI-idiomatic longer AT hint. Any
  value (literal or bound) suppresses.
- `AutomationProperties.LabeledBy` — reference to another element
  that provides the label. Two statically-resolvable forms are
  recognised and scope-validated identically: `{x:Reference <name>}`
  (XAML 2009 primitive) and `{Binding ElementName=<name>}` (classic
  WPF idiom). The target must be declared in the same XAML name scope
  as the button; a dangling reference does not suppress. Other markup
  extensions (`{Binding Path=…}` without `ElementName`,
  `{StaticResource …}`, etc.) suppress permissively because static
  evaluation isn't possible. Non-extension literals suppress on any
  non-empty value.
- `AutomationProperties.IsInAccessibleTree="False"` — explicit
  decorative opt-out (rare for a button — a button without a label
  is usually a design bug). Only a literal `"False"`
  (case-insensitive) or a bound value suppresses; `"True"` reasserts
  default inclusion and therefore does not suppress.
- `AutomationId` — the canonical test-automation hook on UI Automation
  (Windows) and MAUI's `IElement.AutomationId`. Any value (literal or
  bound) suppresses.

```xaml
<!-- non-compliant: glyph button with no AT description -->
<Button Content="&#xE10A;" />

<!-- non-compliant: icon-only image button with no AT description -->
<Button>
    <Image Source="save.png" />
</Button>
```

```xaml
<!-- compliant: verb-first name -->
<Button Content="&#xE10A;" AutomationProperties.Name="Save" />

<!-- compliant: name on the button + decorative image inside -->
<Button AutomationProperties.Name="Save">
    <Image Source="save.png"
           AutomationProperties.IsInAccessibleTree="False" />
</Button>
```

## Interaction with LX0700

`<Button><Image/></Button>` triggers both LX0700 (Image lacks
accessibility description) and LX0704 (Icon button lacks accessibility
description). Both rules are independent — neither suppresses the other —
because the canonical fix names the button and explicitly marks the
inner image decorative. Fixing only one of them leaves the other gap
open: an unnamed button with a named image still announces "button" with
no purpose, and a named button with an audible inner image announces the
image's name in addition to the button's, doubling the announcement.

## What is out of scope

- **Element-form text content.**
  `<Button><TextBlock Text="Save"/></Button>` is ignored —
  `<TextBlock>` is not an icon-class element, so the button is treated
  as text-bearing. Symbolic `Text` on the inner `TextBlock` (e.g.
  `<TextBlock Text="-"/>`) is not inspected; only the button's own
  `Content` attribute is parsed for symbolic values.
- **Custom user-defined icon controls.** The icon-element list is
  fixed (`Image`, `FontIcon`, `SymbolIcon`, `PathIcon`, `BitmapIcon`,
  `ImageIcon`, `Path`). A `<myicons:LogoutGlyph/>` child does not
  trigger the rule — adding open-ended icon-class detection would
  require a project-level list and is deferred.
- **Complex content trees.** A button with two or more layout children
  (e.g. `<Button><StackPanel><Image/><TextBlock/></StackPanel></Button>`)
  is treated as text-bearing. Walking the tree to decide whether the
  combined content is "icon-like" is brittle; assume the author intended
  the visible text to label the button.
- **Button types beyond `Button` and MAUI `ImageButton`.**
  `HyperlinkButton`, `AppBarButton`, `RepeatButton`,
  `ToggleSplitButton`, etc. are not inspected by this rule. Their
  surface differs (`Label`, `Icon`, `Glyph`, `Symbol` properties), and
  the heuristic that fits `Button` does not generalise. Future
  extensions can ship as sibling rules.

## How to fix violations

Add `AutomationProperties.Name="…"` with a verb-first label that names
the button's action ("Save", "Close", "Add to cart", "Open menu"). If an
adjacent Label already names the button, use
`AutomationProperties.LabeledBy="{x:Reference LabelName}"` to reuse it.
On MAUI, prefer `SemanticProperties.Description="…"` for the idiomatic
form.

For the dual-fire `<Button><Image/></Button>` shape, the canonical fix
names the button and marks the inner image decorative:

```xaml
<Button AutomationProperties.Name="Save">
    <Image Source="save.png"
           AutomationProperties.IsInAccessibleTree="False" />
</Button>
```

Marking an icon-only button decorative with `IsInAccessibleTree="False"`
is rare — a button with no label is usually a design bug — so prefer the
name/help-text path.

Test the fix by running a screen reader (Narrator on Windows, TalkBack
on Android, VoiceOver on iOS/macOS) and confirming the announcement
makes sense.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX0704 -->
<Button Content="-" />
```

For a whole file or project:

```json
{ "rules": { "LX0704": "off" } }
```
