# LX0106: Single-child Grid without row or column definitions

<!-- xaml-lint-original; no upstream RXT equivalent. -->

**Applies to:** `Avalonia`  `MAUI`  `Uno`  `UWP`  `WinUI 3`  `WPF`

## Cause

A `<Grid>` has exactly one layout child and no `<Grid.RowDefinitions>` /
`RowDefinitions="…"` and no `<Grid.ColumnDefinitions>` / `ColumnDefinitions="…"`.
The Grid contributes no layout: a `Grid` with no declared rows or columns is a
single-cell container, and a single-cell container around a single child is
structurally redundant — the child can replace the wrapper.

```xaml
<!-- non-compliant: redundant single-cell Grid around one child -->
<Grid>
    <Button Content="OK" />
</Grid>
```

```xaml
<!-- compliant: Grid carries layout intent -->
<Grid>
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto" />
        <RowDefinition Height="*" />
    </Grid.RowDefinitions>
    <Button Grid.Row="0" Content="header" />
</Grid>
```

```xaml
<!-- compliant: replacement for the non-compliant case above -->
<Button Content="OK" />
```

## Rule description

The rule is a local structural heuristic:

- **Layout-child count** — children whose `XName.LocalName` contains a `.` (for example
  `<Grid.Resources>`, `<Grid.Style>`, `<Grid.RowDefinitions>`) are property-element
  pseudo-nodes and do not count. Anything else is a layout child.
- **Declaration presence** — the rule considers both attribute-shorthand syntax
  (`RowDefinitions="…"` / `ColumnDefinitions="…"`) and element-form
  (`<Grid.RowDefinitions><RowDefinition /></Grid.RowDefinitions>`). An **empty**
  `<Grid.RowDefinitions />` does not count as a declaration — it has the same effect
  as no declaration at all.
- **Dialect / framework-version independence.** Presence of the shorthand attribute
  alone suppresses the rule even on `Dialect.Wpf` + `frameworkVersion < 10`, where
  the shorthand is ignored at runtime. LX0104 already flags that scenario — LX0106
  trusts author intent and does not double-flag.
- **No resource-dictionary reach.** A `Style` that sets `RowDefinitions` via a setter
  still fires LX0106 — the rule cannot follow `StaticResource` references.

### What is out of scope

- **Empty `<Grid/>`.** Zero layout children is a different signal — placeholder
  markup, data-bound content that fills later, or `ItemsControl`-style hosts.
  Not flagged.
- **Multi-child Grids.** Two or more children is implicit single-row / single-column
  layout and is legitimate; LX0106 does not flag it.
- **Non-XAML layout intent.** A Grid that exists purely for code-behind to manipulate
  at runtime is indistinguishable from a redundant Grid at lint time. Suppress with
  a pragma if this pattern appears.

## How to fix violations

1. Replace the Grid with the child directly — most common fix.
2. Add the `<Grid.RowDefinitions>` / `<Grid.ColumnDefinitions>` the author intended.
3. Suppress with a pragma when the Grid exists for code-behind reach.

## How to suppress violations

For a single Grid (the next `XElement` after the comment):

```xaml
<!-- xaml-lint disable once LX0106 -->
<Grid>
    <Button Content="OK" />
</Grid>
```

For a block:

```xaml
<!-- xaml-lint disable LX0106 -->
<Grid><Button Content="one" /></Grid>
<Grid><Label Content="two" /></Grid>
<!-- xaml-lint restore LX0106 -->
```

For a whole file or project:

```json
{ "rules": { "LX0106": "off" } }
```

## Default-preset stance

`LX0106` is `DefaultEnabled = false` and `off` in `xaml-lint:recommended`. The rule
is a pure structural heuristic — it fires on Grids that are load-bearing via styles,
visual-tree manipulation from code-behind, or by convention, and those patterns are
common enough that enabling the rule by default would dominate noise over signal.
Projects that want the strict posture can extend `xaml-lint:strict` (where LX0106
is `error`) or enable the rule explicitly.
