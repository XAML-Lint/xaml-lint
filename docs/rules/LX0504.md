# LX0504: Password Entry lacks MaxLength

<!-- Upstream: RXT301. -->

**Applies to:** `MAUI`

## Cause

A MAUI `<Entry>` has `IsPassword="True"` but no `MaxLength` attribute.

## Rule description

Password fields without a `MaxLength` accept unbounded input. That is a poor
UX signal (users don't know when they have reached the system's internal
limit) and a latent security concern — code paths downstream of the field
may assume a bounded string and misbehave on pathological inputs.

The rule fires only when the combination is unambiguous:

- `IsPassword` is a literal `True` (case-insensitive).
- `MaxLength` is absent.

Bound `IsPassword="{Binding …}"` and literal-false `IsPassword="False"` are
both treated as "not a password field right now" and suppress the rule.
Any `MaxLength` (literal or bound) suppresses.

```xaml
<!-- non-compliant: unbounded password input -->
<Entry IsPassword="True" />
```

```xaml
<!-- compliant -->
<Entry IsPassword="True" MaxLength="64" />
```

## How to fix violations

Add a `MaxLength` attribute set to a value appropriate for your backend's
stored-password length limit (commonly 64–128). If you intentionally allow
passphrases without bound, explicitly set `MaxLength` to the framework
maximum to document the decision.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX0504 -->
<Entry IsPassword="True" />
```

For a whole file or project:

```json
{ "rules": { "LX0504": "off" } }
```
