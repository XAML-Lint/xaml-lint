# LX702: TextBox lacks accessibility description

<!-- Upstream: RXT601. -->

## Cause

A `<TextBox>` has no attribute that describes it for assistive technologies.

## Rule description

Screen readers need a name or label to announce a text input's purpose.
On WPF, WinUI 3, UWP, Avalonia, and Uno, a `TextBox` without any of the
suppressors below is opaque to assistive tech — the user hears only
"edit" with no context about what they're editing.

**The rule is off by default in `:recommended`** because most codebases
have not done a deliberate accessibility pass; enabling it there would
flood output. Teams who want to enforce a11y should extend `:strict`, or
explicitly set `LX702: warning` in their config.

**Suppressors:**

- `Name` or `x:Name` — identifies the element in code-behind; screen
  readers can fall back to the runtime name. Any non-empty value
  (literal or bound) suppresses.
- `Header` — WinUI 3 / UWP input label. Any non-empty value (literal
  or bound) suppresses.
- `AutomationProperties.Name` — explicit AT name. Any non-empty value
  (literal or bound) suppresses.
- `AutomationProperties.LabeledBy` — reuses a labelling element.
  Two statically-resolvable element-reference forms are recognised and
  scope-validated identically:
  `{x:Reference <name>}` (XAML 2009 primitive; used on UWP / WinUI 3)
  and `{Binding ElementName=<name>}` (the dominant WPF idiom).
  **The reference target must exist in the same XAML name scope** as
  the `TextBox`. A dangling reference (typo or deleted target) does
  not suppress — it's the bug the rule exists to catch. Other markup
  extensions (`{Binding Path=…}` without `ElementName`,
  `{StaticResource …}`, etc.) still suppress permissively — they
  can't be statically evaluated but represent stated intent.
  Non-extension literals suppress on any non-empty value.
- **Reverse-direction labeling: `<Label Target="{x:Reference <textbox>}">…</Label>`
  or `<Label Target="{Binding ElementName=<textbox>}">…</Label>`** —
  WPF's canonical label-for-input idiom. When any `Label` in the
  document has a `Target` that resolves, in the Label's own name
  scope, to this `TextBox`, the TextBox is considered labeled and
  the rule does not fire. Both markup-extension forms are handled
  symmetrically; a dangling target or a Label inside a `ControlTemplate`
  / `DataTemplate` targeting a name in the outer scope does not
  suppress. Template-scope isolation matches the XAML runtime.

```xaml
<StackPanel xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation">
    <!-- non-compliant: no a11y info -->
    <TextBox />
</StackPanel>
```

```xaml
<StackPanel xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
            xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <!-- compliant: explicit AT name -->
    <TextBox AutomationProperties.Name="Username" />

    <!-- compliant: forward-direction reference to an adjacent label -->
    <Label x:Name="PasswordLabel" Content="Password" />
    <TextBox AutomationProperties.LabeledBy="{x:Reference PasswordLabel}" />

    <!-- compliant: reverse-direction WPF idiom (Label.Target points at the TextBox) -->
    <Label Target="{Binding ElementName=UsernameBox}">_User name:</Label>
    <TextBox Name="UsernameBox" />
</StackPanel>
```

## Known limitation

The rule does not verify the referenced element has user-visible text
(e.g. a `Label.Content` or `TextBlock.Text`). `LabeledBy="{x:Reference SomeBorder}"`
passes even though it's nonsensical for an AT. Tracked separately; the
name-resolution infrastructure is in place but type-aware validation is
not.

## How to fix violations

Pick the lightest suppressor that matches your intent:

- If the `TextBox` is paired with a visible `Label`/`TextBlock`, reuse
  it via `AutomationProperties.LabeledBy="{x:Reference <name>}"` and
  give the label an `x:Name`.
- Otherwise add `AutomationProperties.Name="…"` directly.
- If your workflow already gives every input an `x:Name` for code-behind,
  that alone is enough — screen readers can use the runtime name.

Test the fix by running a screen reader (Narrator on Windows, VoiceOver
on macOS, or your target platform's equivalent) and confirming the
announcement makes sense.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX702 -->
<TextBox />
```

For a whole file or project:

```json
{ "rules": { "LX702": "off" } }
```
