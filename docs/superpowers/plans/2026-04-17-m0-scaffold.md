# M0 — Scaffold Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the repo skeleton — solution file, all csproj projects, shared MSBuild infrastructure, release-tracking log files, and cross-platform CI — so M1 can start adding engine code without re-scaffolding.

**Architecture:** Seven csproj projects referenced from a single `.slnx` solution. Shared MSBuild properties live in `Directory.Build.props`; package versions are centrally managed in `Directory.Packages.props`; `Nerdbank.GitVersioning` derives SemVer from git history. Four source projects (`XamlLint.Core`, `XamlLint.Core.SourceGen`, `XamlLint.DocTool`, `XamlLint.Cli`) under `src/`, three test projects under `tests/` wired as xUnit v3 + Microsoft Testing Platform self-hosting executables. GitHub Actions CI runs `dotnet build` + `dotnet test` on Windows, Linux, and macOS.

**Tech Stack:** .NET 10 SDK, `.slnx` solution format, `Nerdbank.GitVersioning`, Central Package Management (CPM), xUnit v3 + Microsoft Testing Platform (MTP), AwesomeAssertions (MIT fork of FluentAssertions), FakeItEasy + FakeItEasy.Analyzer.CSharp for fakes, Roslyn `IIncrementalGenerator` (for the source-gen project, which stays an empty stub until M1).

## Notes before starting

**Deviation from design spec §10.** The M0 bullet says "three csproj projects building clean", but design §9 enumerates **four** src projects (`XamlLint.Core`, `XamlLint.Core.SourceGen`, `XamlLint.DocTool`, `XamlLint.Cli`) and **three** test projects (`XamlLint.Core.Tests`, `XamlLint.Cli.Tests`, `XamlLint.Plugin.Tests`) — seven total. The "three" was an older estimate. This plan scaffolds all seven so M1 doesn't spend time creating more csproj files before it can write its first engine diagnostic.

**Commit cadence.** One commit per task (9 commits). Every commit must leave the repo building clean on `dotnet build xaml-lint.slnx`.

**Package version pinning.** Version numbers in this plan are best-known-good as of 2026-04-17. Each task that introduces new packages has a verification step to run `dotnet package search <name> --take 1` and bump to the latest listed version if newer — then pin that exact version in `Directory.Packages.props`. Do not leave floating versions.

**Worktree.** Run this plan in a fresh git worktree (`git worktree add ../xaml-lint-m0 -b m0-scaffold`) so the scaffolding doesn't collide with the existing `main` branch. Or work directly on a feature branch off `main`. Either way, don't work on `main` itself.

**Files that already exist and must be preserved.** `README.md`, `LICENSE` (has Matt Lacey's copyright — do not touch), `CLAUDE.md`, `.gitignore`, `.claude/settings.json`, `docs/superpowers/specs/2026-04-17-xaml-lint-design.md`.

## Task 3: XamlLint.Core.SourceGen (analyzer project)

Adds the Roslyn `IIncrementalGenerator` project and wires it into `XamlLint.Core` as an analyzer reference. The generator itself is empty in M0 — M1 adds the rule-catalog generator logic. But the reference must be present now so build semantics (analyzer loading, `EnforceExtendedAnalyzerRules`) are already in place.

**Files:**
- Create: `src/XamlLint.Core.SourceGen/XamlLint.Core.SourceGen.csproj`
- Modify: `src/XamlLint.Core/XamlLint.Core.csproj` (add project reference as analyzer)
- Modify: `xaml-lint.slnx` (add new project)

- [ ] **Step 1: Create the source-generator csproj**

Create `src/XamlLint.Core.SourceGen/XamlLint.Core.SourceGen.csproj`:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFramework>netstandard2.0</TargetFramework>
    <IsRoslynComponent>true</IsRoslynComponent>
    <EnforceExtendedAnalyzerRules>true</EnforceExtendedAnalyzerRules>
    <IsPackable>false</IsPackable>
    <ImplicitUsings>disable</ImplicitUsings>
    <ManagePackageVersionsCentrally>false</ManagePackageVersionsCentrally>
    <Description>Build-time source generator for xaml-lint's rule catalog. Not shipped as a standalone package.</Description>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.CodeAnalysis.Analyzers" Version="5.3.0">
      <PrivateAssets>all</PrivateAssets>
      <IncludeAssets>runtime; build; native; contentfiles; analyzers; buildtransitive</IncludeAssets>
    </PackageReference>
    <PackageReference Include="Microsoft.CodeAnalysis.CSharp" Version="5.3.0" PrivateAssets="all" />
  </ItemGroup>

</Project>
```

Why these settings:
- `netstandard2.0` — Roslyn source generators must load into older MSBuild hosts; this is a hard requirement, not a preference.
- `IsRoslynComponent=true` — enables source-generator tooling (debugging attach, build integration).
- `EnforceExtendedAnalyzerRules=true` — forbids APIs that break in the analyzer host (e.g., `Environment.NewLine`). Pays off once the generator has code.
- `ImplicitUsings=disable` — `netstandard2.0` ImplicitUsings imports a subset that differs from net10; keeping generator code tight avoids surprises.
- `ManagePackageVersionsCentrally=false` — source-generator projects have had historical friction with CPM (version flow through the analyzer packaging path breaks). This project opts out and pins its Roslyn versions inline. Every other project in the solution stays on CPM.
- `Microsoft.CodeAnalysis.CSharp` with `PrivateAssets="all"` — the generator DLL itself doesn't ship Roslyn; consumers get Roslyn from whoever loads the analyzer.
- `Microsoft.CodeAnalysis.Analyzers` with the full `<IncludeAssets>` list — meta-analyzers that check generator/analyzer code for common bugs (`RS1001`…). The explicit `analyzers` asset is the one that matters here; without it the meta-analyzer DLLs are masked by the default transitive-asset rules for analyzer packages, and the checks never fire at build time.
- No `Nerdbank.GitVersioning` reference — source generators are build-time tools whose assembly version doesn't matter, and keeping it out avoids needing another inline-version pin in this opt-out project.

- [ ] **Step 2: Wire Core → SourceGen as an analyzer reference**

Edit `src/XamlLint.Core/XamlLint.Core.csproj` — add a second `<ItemGroup>` after the existing one:

```xml
  <ItemGroup>
    <ProjectReference Include="..\XamlLint.Core.SourceGen\XamlLint.Core.SourceGen.csproj"
                      OutputItemType="Analyzer"
                      ReferenceOutputAssembly="false" />
  </ItemGroup>
```

`OutputItemType="Analyzer"` tells MSBuild to pass the generator DLL to the compiler as an analyzer.
`ReferenceOutputAssembly="false"` means `XamlLint.Core` does not consume the generator's public API as a runtime dependency — the generator only emits code during Core's compilation.

Edit `xaml-lint.slnx`:

```xml
  </Folder>

```

Expected: both projects build clean. No analyzer warnings (the generator has no code yet, so there's nothing to analyze).

## Task 9: CI workflow + verify green

Adds GitHub Actions workflow running `dotnet build` and `dotnet test` on Windows, Linux, and macOS against .NET 10. This is the M0 exit gate — M0 is only "done" once CI is green on all three OSes for a fresh commit.

**Files:**
- Create: `.github/workflows/ci.yml`

- [ ] **Step 1: Create the CI workflow**

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    name: ${{ matrix.os }}
    runs-on: ${{ matrix.os }}

    strategy:
      fail-fast: false
      matrix:
        os: [windows-latest, ubuntu-latest, macos-latest]

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Nerdbank.GitVersioning needs full history

      - name: Setup .NET SDK
        uses: actions/setup-dotnet@v4
        with:
          global-json-file: global.json

      - name: Print SDK version
        run: dotnet --info

      - name: Restore
        run: dotnet restore xaml-lint.slnx

      - name: Build
        run: dotnet build xaml-lint.slnx --configuration Release --no-restore

      - name: Test
        run: dotnet test xaml-lint.slnx --configuration Release --no-build --verbosity normal
```

Notes:
- `fetch-depth: 0` — Nerdbank.GitVersioning reads the commit height from git log; a shallow clone computes the wrong version.
- `global-json-file: global.json` — pins the CI SDK to whatever `global.json` specifies. No drift between local and CI.
- `fail-fast: false` — one OS failing shouldn't cancel the others; seeing all three outcomes at once speeds debugging.
- Matrix is only `os`, not `os × tfm`. Adding more TFMs later is trivial; starting with one keeps CI runtime small.

- [ ] **Step 2: Commit the workflow**


Common failure modes and fixes:
- **`dotnet restore` fails on one OS only** — a package has a missing platform-specific binary. Check the affected package's NuGet page for TFM support.
- **`dotnet build` fails with "SDK not found"** — `global.json` pins an SDK version not available in `setup-dotnet`'s index yet. Loosen the pin (`"rollForward": "latestMinor"`) or bump to a more widely-available version.
- **`dotnet test` reports `Passed: 0`** — MTP didn't discover any tests. Inspect the test project's csproj: `EnableMicrosoftTestingPlatform`, `UseMicrosoftTestingPlatformRunner`, `TestingPlatformDotnetTestSupport` must all be `true` and `OutputType` must be `Exe`.

6. `AnalyzerReleases.Shipped.md` and `AnalyzerReleases.Unshipped.md` both exist.
7. `LICENSE`, `README.md`, `CLAUDE.md`, and the existing design spec are unchanged.

No user-visible release is cut from M0 — it is a pre-M1 gate, not a tag. M1 will be the first `v0.1.0-alpha.*` prerelease.
