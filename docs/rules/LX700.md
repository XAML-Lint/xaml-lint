# LX700: Image lacks accessibility description

<!-- Upstream: RXT350. -->

## Cause

A MAUI `<Image>` has no attribute that describes it for assistive
technologies, and is not explicitly marked as decorative.

## Rule description

Screen readers cannot interpret an image's visual content. For images that
convey meaning (an icon with semantic weight, a photograph that communicates
information), the app must provide text alternatives through MAUI's
`AutomationProperties` attached properties. For images that are purely
decorative (a divider, a background flourish), the app should explicitly
opt out of the accessibility tree so the screen reader doesn't announce
an anonymous "image" to the user.

**The rule is off by default in `:recommended`** because most codebases
have not done a deliberate accessibility pass; enabling it there would
flood output. Teams who want to enforce a11y should extend `:strict`, or
explicitly set `LX700: warning` in their config.

**Suppressors:**

- `AutomationProperties.Name` — concise text alternative. Any value
  (literal or bound) suppresses.
- `AutomationProperties.HelpText` — longer description. Any value
  (literal or bound) suppresses.
- `SemanticProperties.Description` — MAUI-idiomatic short AT name. Any
  value (literal or bound) suppresses.
- `SemanticProperties.Hint` — MAUI-idiomatic longer AT hint. Any value
  (literal or bound) suppresses.
- `AutomationProperties.LabeledBy` — reference to another element that
  provides the label. Two statically-resolvable forms are recognised
  and scope-validated identically: `{x:Reference <name>}` (XAML 2009
  primitive) and `{Binding ElementName=<name>}` (classic WPF idiom —
  rare on MAUI but syntactically valid). The target must be declared
  in the same XAML name scope as the image; a dangling reference
  does not suppress. Other markup extensions (`{Binding Path=…}`
  without `ElementName`, `{StaticResource …}`, etc.) suppress
  permissively because static evaluation isn't possible. Non-extension
  literals suppress on any non-empty value.
- `AutomationProperties.IsInAccessibleTree="False"` — explicit decorative
  opt-out. Only a literal `"False"` (case-insensitive) or a bound value
  suppresses; `"True"` reasserts default inclusion and therefore does not
  suppress — the image is in the accessibility tree and still needs a name.

```xaml
<ContentPage xmlns="http://schemas.microsoft.com/dotnet/2021/maui">
    <!-- non-compliant: no a11y info -->
    <Image Source="icon.png" />
</ContentPage>
```

```xaml
<ContentPage xmlns="http://schemas.microsoft.com/dotnet/2021/maui">
    <!-- compliant: meaningful image has a name -->
    <Image Source="icon.png" AutomationProperties.Name="Home" />

    <!-- compliant: decorative image explicitly excluded -->
    <Image Source="divider.png" AutomationProperties.IsInAccessibleTree="False" />
</ContentPage>
```

## Known limitation

The rule does not verify the element referenced by
`AutomationProperties.LabeledBy="{x:Reference …}"` has user-visible text
(e.g. a `Label.Text` or adjacent caption). A reference to a layout
`<Border/>` passes the check even though it's nonsensical for an AT.
Tracked separately; the name-resolution infrastructure is in place but
type-aware validation is not.

## How to fix violations

Decide whether the image is meaningful or decorative:

- **Meaningful**: add `AutomationProperties.Name="…"` (short) and/or
  `AutomationProperties.HelpText="…"` (longer). If an adjacent Label
  already names the image, use
  `AutomationProperties.LabeledBy="{x:Reference LabelName}"` to reuse it.
- **Decorative**: add `AutomationProperties.IsInAccessibleTree="False"`.
  The screen reader will skip it entirely.

Test the fix by running a screen reader (TalkBack on Android, VoiceOver on
iOS) and confirming the announcement makes sense.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX700 -->
<Image Source="icon.png" />
```

For a whole file or project:

```json
{ "rules": { "LX700": "off" } }
```
