# LX0302: Unused x:Name

**Applies to:** `Avalonia`  `MAUI`  `Uno`  `UWP`  `WinUI 3`  `WPF`

## Cause

An `x:Name` attribute declares a namescope identifier that no other XAML in the same file
references. Either the name is a leftover from an earlier iteration of the markup and can
be deleted, or the element is reached only from C# code-behind (in which case the rule
cannot see the reference and may be a false positive — see "Default-preset stance" below).

## Rule description

The rule walks two passes over the document:

1. **Reference scan.** Every attribute is inspected for:
   - Markup-extension reference forms: `{Binding ElementName=X}`, `{x:Reference X}`,
     `{x:Reference Name=X}`, `{Reference X}`.
   - Literal-attribute reference forms: `TargetName="X"`, `SourceName="X"`, and the
     attached-property equivalents (`Storyboard.TargetName="X"`, `Setter.TargetName="X"`,
     `Trigger.SourceName="X"`, `Condition.SourceName="X"`, `EventTrigger.SourceName="X"`).

   Each reference is resolved against the reference-site element's XAML name scope using
   the same scope-aware index LX0202 / LX0203 use. A reference that resolves marks the
   declaring element as "used".

2. **Declaration scan.** Every element carrying an `x:Name` attribute in the XAML 2006
   (`http://schemas.microsoft.com/winfx/2006/xaml`) or XAML 2009
   (`http://schemas.microsoft.com/winfx/2009/xaml`) namespace is checked against the used
   set. Elements not in the set are flagged. Unprefixed `Name="X"` declarations are
   ignored — the unprefixed form doubles as a framework-level property assignment and is
   out of scope here (same split as LX0300).

XAML namescopes follow template boundaries — `ControlTemplate`, `DataTemplate`,
`ItemsPanelTemplate`, and `HierarchicalDataTemplate` each open a nested scope that is
isolated from the outer scope. A name declared inside a template and referenced only
from outside the template (or vice versa) is reported as unused, because the runtime
cannot resolve across that boundary.

```xaml
<!-- non-compliant: 'Orphan' is declared but nothing in this file references it -->
<StackPanel xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
            xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <Label x:Name="Orphan" />
    <TextBox />
</StackPanel>
```

```xaml
<!-- compliant: 'Header' is referenced via {Binding ElementName=...} -->
<StackPanel xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
            xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <Label x:Name="Header" />
    <TextBox Text="{Binding ElementName=Header, Path=Content}" />
</StackPanel>
```

```xaml
<!-- compliant: 'Chrome' is referenced via a trigger Setter.TargetName -->
<ControlTemplate xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                 xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
                 TargetType="Button">
    <Border x:Name="Chrome">
        <ContentPresenter />
    </Border>
    <ControlTemplate.Triggers>
        <Trigger Property="IsPressed" Value="True">
            <Setter TargetName="Chrome" Property="Background" Value="Red" />
        </Trigger>
    </ControlTemplate.Triggers>
</ControlTemplate>
```

### What is out of scope

- **Code-behind references.** The rule does not parse C#. A name reached only from code
  (`this.Header.Content = ...`, `var x = Header;`) will be reported as unused. This is
  the main source of false positives and the reason the rule is off by default.
- **`{x:Bind path.Name}` typed paths.** `x:Bind` paths resolve against typed members, not
  named XAML elements, and are intentionally not parsed.
- **Nested markup extensions.** Only the outer attribute value is inspected. An
  `ElementName=` buried inside a nested converter or source expression
  (`{Binding Source={x:Reference Foo}}`) is not counted as a reference.
- **Unprefixed `Name="X"` declarations.** The declaration side of this rule is `x:Name`
  only. Elements carrying only an unprefixed `Name=` attribute are never flagged, because
  the unprefixed form doubles as a framework-level property value whose removal could
  break runtime behaviour.

## How to fix violations

1. Delete the `x:Name` if the element genuinely doesn't need to be reached.
2. If the element is referenced only from code-behind, suppress locally with a pragma —
   this is the expected false-positive pattern on the default-off rule.
3. If the element is referenced via `x:Bind` or a converter-embedded extension, suppress
   at the file or project level and leave the rule off for that project.

## How to suppress violations

For a single element (the next `XElement` after the comment):

```xaml
<!-- xaml-lint disable once LX0302 -->
<Label x:Name="UsedFromCodeBehind" />
```

For a block of markup:

```xaml
<!-- xaml-lint disable LX0302 -->
<Label x:Name="One" />
<Label x:Name="Two" />
<!-- xaml-lint restore LX0302 -->
```

For a whole file or project:

```json
{ "rules": { "LX0302": "off" } }
```

## Default-preset stance

`LX0302` is `DefaultEnabled = false` and `off` in `xaml-lint:recommended`. The rule only
sees XAML references; projects that wire named elements from code-behind (the common WPF
pattern) will see false positives on every code-behind-only name. Teams that want the
cleanup signal can extend `xaml-lint:strict` (where LX0302 is `warning`) or enable the
rule explicitly in their config.
