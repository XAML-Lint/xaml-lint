# LX503: Entry lacks Keyboard

<!-- Upstream: RXT300. -->

**Applies to:** `MAUI`

## Cause

A MAUI `<Entry>` element is declared without a `Keyboard` attribute.
`Keyboard` is the hint the platform uses to select a soft keyboard layout
(numeric, email, URL, telephone, …).

## Rule description

The default `Keyboard` is `Default`, which yields a general-purpose keyboard.
That is rarely the right thing for fields that expect numbers, email
addresses, URLs, or phone numbers — users have to switch to the numeric or
symbol subpage themselves, and autocorrect happily "corrects" a URL into a
sentence.

Any `Keyboard` value — literal (`"Numeric"`, `"Email"`, `"Url"`, …) or
bound (`{Binding Kbd}`) — is accepted. The rule does not try to validate
the value; its job is to flag the absence.

Only unprefixed `<Entry>` elements are considered.

```xaml
<!-- non-compliant: defaults to the general-purpose keyboard -->
<Entry />
```

```xaml
<!-- compliant: soft keyboard opens in numeric layout -->
<Entry Keyboard="Numeric" />
```

## How to fix violations

Add a `Keyboard` attribute with the most specific value that matches the
field's semantics. Common choices:

- `Numeric` — integers and decimals
- `Email` — email input
- `Url` — URLs
- `Telephone` — phone numbers
- `Text` — explicit "general text" (silences the rule without behavior change)
- `Chat` — freeform messaging input

If no specific keyboard fits and the field genuinely is freeform prose,
use `Text` explicitly to document the decision.

## How to suppress violations

For a single element:

```xaml
<!-- xaml-lint disable once LX503 -->
<Entry />
```

For a block:

```xaml
<!-- xaml-lint disable LX503 -->
<Entry />
<Entry />
<!-- xaml-lint restore LX503 -->
```

For a whole file or project:

```json
{ "rules": { "LX503": "off" } }
```
